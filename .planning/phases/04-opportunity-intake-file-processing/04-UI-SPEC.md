### Design Thesis
- **Visual thesis:** continue the Phase 3 dark workflow-console baseline and turn Overview into a focused intake desk, not a generic form page and not an upload-marketing surface.
- **Content thesis:** keep `source material` and `minimum opportunity context` visibly separate so users always understand what is original input versus editable business framing.
- **Interaction thesis:** the page must feel draft-preserving and operational: visible save trust signals, explicit file-state feedback, and one clear next step with precise blocked messaging.

Additional copy rules:
- Use utility copy only. This page is an intake workspace, not a product tour.
- Every blocked state must name the exact missing condition: missing intake details, missing usable source text, file still processing, or save failure blocking trust.
- `Generate lead brief` should read as momentum, not risk. The adjacent helper text must state why it is available or what still blocks it.
- File-state copy must always pair status with next step. Example patterns: `Processing your PDF. Keep editing raw input while extraction runs.` and `Processing failed. Retry extraction or continue with manual source material.`
- Do not use vague button labels such as `Continue` or `Proceed` when the destination is Phase 5.

## Opportunity Shell Contract

### Top-Level Hierarchy
1. Opportunity header with title, company, owner/context metadata, and status chip
2. Stable five-step workflow stepper: `Lead Intake`, `Lead Brief`, `Discovery`, `Proposal Draft`, `Follow-up`
3. Intake working surface with the source workspace on the left and the minimum-context workspace on the right
4. Recovery/help notices below the working surface

### Shell Rules
- Preserve the Phase 3 global customer shell and navigation without visual reset.
- The page eyebrow should shift from generic command-center framing to opportunity-specific framing: `Opportunity intake`.
- The stepper must remain visible above long content and must not be buried under the raw-input editor.
- The current step uses accent styling. Future steps use muted labels. Blocked downstream steps use muted text plus textual reason, never accent.
- Header actions must keep `Generate lead brief` visually dominant and `Save opportunity` secondary.

### Desktop and Laptop Layout
- Primary working layout is a two-column desk at `1280px` and `1440px`.
- Use an asymmetric split: `source workspace` at approximately `1.5fr` and `minimum context` at approximately `0.95fr`.
- At `1024px`, keep the source workspace first and move the context workspace below it; this is still a laptop layout, not a mobile redesign.
- No horizontal overflow is allowed at `1024px`.

## Working Surface Contract

### Left Column: Source Workspace
- Order is fixed: `Raw input` editor, `PDF upload and file state`, then `Extracted text preview`.
- `Raw input` is the primary source editor and must support long text comfortably. Minimum visible height: `320px`.
- Upload must support click-to-select and drag-and-drop, but drag-and-drop cannot be the only path.
- The file module must stay visible while the user edits other fields; do not hide it behind a modal or toast.
- `Extracted text preview` appears inline in the left column when a file is `ready`. Minimum visible height: `220px` with internal scroll if needed.

### Right Column: Minimum Opportunity Context
- Fixed field set for Phase 4: `title`, `company_name`, `contact_name`, `contact_email`, `requested_service`, visible `source_type`, and owner as non-editable context metadata.
- `Title` may show a suggestion chip or prefill before manual editing begins. Once the user edits the field, no background auto-rename is allowed.
- `contact_email` validates only when present.
- The right column ends with the action area, not with unrelated metadata blocks.
- Do not add CRM-like sidebars, assignment tools, activity feeds, or analytics modules.

### Action Area
- Primary action: `Generate lead brief`
- Secondary action: `Save opportunity`
- Support line directly under the actions:
  - when ready: explain that Phase 5 will open next
  - when blocked: name the missing requirement in one sentence
  - when save is pending/failed: surface persistence trust state before generation

## File State Contract

These states must be visible in-page and distinct in text, iconography, and color:

### `uploaded`
- Show filename, uploaded time, and queued/extraction-starting copy.
- Use neutral/secondary surface styling with a clear state label.
- Keep manual editing fully available.

### `processing`
- Show a progress treatment that feels active but calm: spinner or progress bar plus status text.
- Status copy must explicitly tell the user they can keep editing raw input and minimum fields.
- Treat this as informational/warning, not as an error.
- A processing file alone does not unlock generation.

### `ready`
- Show success styling plus extracted preview availability.
- Surface extraction freshness metadata: `Ready`, timestamp, and file name.
- Provide a quiet affordance to review extracted text before generating.

### `failed`
- Use warning/failure styling that is stronger than `processing` but not destructive red.
- Show the failure state inline with a visible `Retry extraction` action.
- Keep manual raw-input editing and save behavior available.
- If failure details exist, show them in compact body copy; do not collapse failure into a badge.

### Retry Rules
- Retry must be explicit and framed as a new processing attempt.
- Retry UI stays local to the file module; do not redirect the user or block the entire page.
- Previous failure context may remain visible until the new attempt starts, but the page should clearly indicate that a fresh attempt is underway.

## Generation Gate Contract

- `Generate lead brief` is enabled only when:
  - `title` is present
  - `company_name` is present
  - `requested_service` is present
  - and either `raw_input` contains meaningful text or extracted text is available from a `ready` file
- A file in `processing` does not satisfy the source-text requirement.
- A failed or absent file does not block generation if manual source text is sufficient.
- Disabled CTA without explanation is forbidden.

### Blocked Messaging
- Missing business context: `Add the core opportunity details before generating the lead brief.`
- Missing usable source text: `Add raw source notes or wait for extracted text before generating the lead brief.`
- File still processing with no manual text: `Your PDF is still processing. You can wait for extraction or add manual source notes now.`
- Save trust issue: `We couldn't save the latest intake changes. Retry save before generating the lead brief.`

### Success Handoff
- Successful generation navigates directly into the Lead Brief workspace.
- The overview page should not leave the user at rest after success or require them to choose the next route manually.

## Save and Persistence Contract

- The page behaves as draft-preserving by default with low-friction autosave or equivalent silent persistence.
- Show a quiet persistence indicator near the action area: `Savingâ€¦`, `Saved just now`, or `Save failed`.
- Save feedback must be inline on the page. Toast-only persistence is not acceptable.
- `Save opportunity` exists as a trust anchor and retry action, not as the only way to persist.
- Save errors must attach to the working surface or action area and preserve user-entered text.

## Shared State Contract

The overview page must support these states visually and semantically:
- `loading`
- `empty`
- `error`
- `blocked`
- `retry`
- `success`
- file-specific `uploaded`
- file-specific `processing`
- file-specific `ready`
- file-specific `failed`

Rules:
- Opportunity-level states use the existing `ProductStateBlock` pattern or a direct extension of it.
- File-level states live inside the file module and must not replace the rest of the page unless the entire record is unavailable.
- `blocked` and `error` copy must not collapse into the same language.
- Empty states must still show the working layout and a clear next action where possible.

## Interaction and Motion Contract

- Motion budget: `150msâ€“220ms`
- Hover may adjust color, border, and shadow only; no card-lift choreography and no scale transforms that shift layout.
- Autosave, file-processing, and retry transitions must feel calm and operational, not flashy.
- File state changes may animate icon rotation, opacity, or progress fill, but must respect `prefers-reduced-motion`.
- Keep the stepper and action bar visually stable during async transitions.
